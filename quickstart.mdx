Choose the deployment method that matches your environment. Docker Compose is the fastest path for local development. Docker Swarm and Kubernetes are available for cluster deployments.

## Docker Compose (local development)

Docker Compose builds the images from source and mounts local directories as volumes so code changes are reflected immediately without rebuilding.

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/dockersamples/example-voting-app.git
    cd example-voting-app
    ```
  </Step>

  <Step title="Build and start all services">
    Run the following command from the project root:

    ```bash
    docker compose up
    ```

    Compose builds the `vote`, `result`, and `worker` images, pulls `redis:alpine` and `postgres:15-alpine`, and starts all five services. Health checks on Redis and PostgreSQL gate the dependent services so they start in the correct order.
  </Step>

  <Step title="Open the app">
    | Interface | URL |
    |-----------|-----|
    | Vote | [http://localhost:8080](http://localhost:8080) |
    | Results | [http://localhost:8081](http://localhost:8081) |

    Cast a vote in the vote interface and watch the results update in real time in the results interface.
  </Step>
</Steps>

### Customise the vote options

By default the two options are **Cats** and **Dogs** (set in `vote/app.py`). Override them with environment variables before starting:

```bash
OPTION_A=Tabs OPTION_B=Spaces docker compose up
```

Or add them to a `.env` file in the project root:

```bash
OPTION_A=Tabs
OPTION_B=Spaces
```

### Seed the database with test votes

A dedicated `seed` service is included but disabled by default. Run it with the `seed` profile to populate the database with sample votes:

```bash
docker compose --profile seed up -d
```

The seed service exits after it finishes — it has `restart: "no"` and depends on the `vote` service being healthy before it runs.

### Stop and remove the app

To stop all containers:

```bash
docker compose down
```

To also remove the named `db-data` volume (discards all stored votes):

```bash
docker compose down --volumes
```

<Tip>
  The `result` service also exposes a Node.js debugger on `127.0.0.1:9229` when running under Compose, so you can attach a debugger from your editor.
</Tip>

## Docker Swarm

The `docker-stack.yml` file is configured specifically for Swarm. It uses pre-built images from Docker Hub and runs two replicas each of the `vote` and `worker` services.

<Warning>
  Do not use `docker-stack.yml` with `docker compose up`. Multiple replicas trying to bind to the same host port will fail outside of Swarm mode.
</Warning>

<Steps>
  <Step title="Initialise a swarm">
    If you do not already have a swarm, create one:

    ```bash
    docker swarm init
    ```
  </Step>

  <Step title="Deploy the stack">
    From the project root, deploy the stack using `docker-stack.yml`:

    ```bash
    docker stack deploy --compose-file docker-stack.yml vote
    ```

    This creates the stack named `vote` with all five services.
  </Step>

  <Step title="Check service status">
    Verify all services are running:

    ```bash
    docker stack services vote
    ```

    Wait until all replicas report as running before opening the app.
  </Step>

  <Step title="Open the app">
    | Interface | URL |
    |-----------|-----|
    | Vote | [http://localhost:8080](http://localhost:8080) |
    | Results | [http://localhost:8081](http://localhost:8081) |
  </Step>
</Steps>

### Service replicas

The Swarm stack runs the following replicas:

| Service | Replicas |
|---------|----------|
| `vote` | 2 |
| `worker` | 2 |
| `result` | 1 |
| `redis` | 1 |
| `db` | 1 |

### Remove the stack

```bash
docker stack rm vote
```

To leave the swarm entirely:

```bash
docker swarm leave --force
```

## Kubernetes

The `k8s-specifications/` directory contains Kubernetes Deployment and Service YAML manifests for all five services.

<Steps>
  <Step title="Ensure your cluster context is set">
    Verify `kubectl` is pointing at the cluster you want to use:

    ```bash
    kubectl config current-context
    ```
  </Step>

  <Step title="Create all deployments and services">
    Apply all manifests in the `k8s-specifications/` directory:

    ```bash
    kubectl create -f k8s-specifications/
    ```

    Resources are created in the current namespace (`default` if you have not changed it).
  </Step>

  <Step title="Check pod status">
    Wait for all pods to reach the `Running` state:

    ```bash
    kubectl get pods
    ```
  </Step>

  <Step title="Open the app">
    The services are exposed as NodePort services on each host in the cluster:

    | Interface | Port |
    |-----------|------|
    | Vote | `31000` |
    | Results | `31001` |

    Access them at `http://<node-ip>:31000` and `http://<node-ip>:31001`.
  </Step>
</Steps>

### Remove the app

Delete all resources created from the manifests:

```bash
kubectl delete -f k8s-specifications/
```

<Note>
  Kubernetes deployments use NodePort services on ports 31000 and 31001, which differ from the 8080/8081 ports used in Compose and Swarm deployments.
</Note>
