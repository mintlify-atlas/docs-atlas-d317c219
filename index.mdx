## What is the Example Voting App?

The Example Voting App is a reference distributed application maintained by Docker. It is intentionally simple — its purpose is to demonstrate how to connect multiple containerized services together using Docker tooling, not to serve as a production-ready application template.

The app lets users vote between two options (default: Cats vs Dogs) and view live results. Votes are submitted through a Python web app, queued in Redis, processed by a .NET worker, stored in PostgreSQL, and displayed in real time by a Node.js results app.

## Key concepts demonstrated

<CardGroup cols={3}>
  <Card title="Multi-container networking" icon="network-wired">
    Services communicate over isolated Docker bridge networks with front-tier and back-tier separation.
  </Card>
  <Card title="Message queue pattern" icon="list">
    The vote service publishes to a Redis list; the worker consumes it asynchronously.
  </Card>
  <Card title="Persistent storage" icon="database">
    PostgreSQL uses a named Docker volume so vote data survives container restarts.
  </Card>
  <Card title="Health checks" icon="heart-pulse">
    Redis and Postgres expose shell-based health checks so dependent services wait for readiness.
  </Card>
  <Card title="Real-time updates" icon="bolt">
    The result service uses Socket.IO to push live vote counts to the browser without polling.
  </Card>
  <Card title="Multi-platform deployment" icon="layer-group">
    The same application runs on Docker Compose, Docker Swarm, and Kubernetes.
  </Card>
</CardGroup>

## Application stack

| Service | Technology | Role |
|---------|-----------|------|
| vote | Python / Flask | Front-end voting interface |
| result | Node.js / Express | Real-time results display |
| worker | .NET / C# | Vote queue consumer |
| redis | Redis (alpine) | In-memory vote queue |
| db | PostgreSQL 15 | Persistent vote storage |
